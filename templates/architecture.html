System Architecture

Overview

The Dual Model IDS Web System - Burak Doƒüan implements a two-stage attack detection pipeline with confidence-based routing. Network traffic is first classified as benign or attack, then attack type is identified. Confidence scores determine the response actions (blocking, alerting, logging).

Processing Pipeline

1
Data Upload: Network traffic CSV files are uploaded through the web interface
2
Preprocessing: Data is loaded, validated, and prepared for feature extraction
3
Stage 1 - Binary Classification: Determines if traffic is benign or attack
4
Stage 2 - Multiclass Classification: If attack detected, identifies attack type
5
Confidence Routing: Confidence scores determine response actions
6
Results Display: Visualize predictions with charts and detailed statistics

How Attack Type Detection Works

After Stage 1 determines if traffic is an "Attack" or "Benign", the system uses confidence levels to decide whether to run Stage 2 (attack type classification). Here's how the decision works based on the confidence score from Stage 1:

HIGH Confidence (>90%)
Decision: "This is DEFINITELY an attack" - We are very confident
Stage 2 Runs? ‚úÖ YES - Immediately
Logic: Since we're very confident it's an attack (>90%), we run Stage 2 to identify exactly what type of attack it is (DDoS, Reconnaissance, DoS, MITM, etc.). The attack is marked as "CONFIRMED".
Result: ‚úì attack_type = "DDoS" (or other type)
‚úì attack_confidence = "CONFIRMED" (not possible)
‚úì confidence_score = 92.8% (Stage 1 confidence)
Classification is CONFIRMED - take immediate action
Action: BLOCK immediately | Generate P1 ticket | Alert SOC team | Update firewall rules in real-time
MEDIUM Confidence (70-90%)
Decision: "This might be an attack" - We are moderately confident, but not 100% sure
Stage 2 Runs? ‚úÖ YES - But Marked as "POSSIBLE"
Logic: We think it's an attack, so we run Stage 2 to determine the attack type. However, the uncertainty means the attack classification is marked as "POSSIBLE" rather than "CONFIRMED". Additional verification needed before taking destructive actions.
Result: ‚úì attack_type = "Reconnaissance" (or other type)
‚úì attack_confidence = "POSSIBLE" (not confirmed)
‚úì confidence_score = 78% (Stage 1 confidence)
Classification is SUSPICIOUS - needs investigation before action
Action: INVESTIGATE | Add to analyst queue (P2) | Enable rate limiting | Run in sandbox | Enrich with threat intelligence
LOW Confidence (<70%)
Decision: "This is probably normal traffic" - We are not confident it's an attack
Stage 2 Runs? ‚ùå NO - Skipped
Logic: We don't think it's an attack (confidence <70%), so there's no point running Stage 2 to classify attack types. We assume the traffic is safe and treat it as normal, benign network activity.
Result: ‚úó attack_type = None (not determined)
‚úó attack_probability = None (no Stage 2 run)
Classification is SAFE - no threat identified
Action: ALLOW | Log routine activity | No alerts | No blocking | Normal operations continue

Confidence-Based Routing

The system uses three confidence levels to determine response actions:

HIGH (>90%)

  • P1 incident ticket
  • SOC notification
MEDIUM (70-90%)
  • Rate limiting

LOW (<70%)

  • Sampling
  • Retraining dataset
  • No impact

Machine Learning Models - Best Performers Selected

Model Selection Process

The system was trained with 6 different machine learning models for each classification task. Performance comparison testing identified the best-performing model for each stage:

Stage 1: Binary Classification (Attack vs Benign)

Models Tested: Logistic Regression, MLP, Random Forest, LightGBM, K-Nearest Neighbors, Gaussian Naive Bayes

‚úì Selected Model: Random Forest

Reason: Best accuracy and robustness in distinguishing between benign and attack traffic. Handles feature interactions well and provides stable predictions across diverse network patterns.

Stage 2: Multiclass Classification (Attack Type Identification)

Models Tested: Logistic Regression, MLP, Random Forest, LightGBM, K-Nearest Neighbors, Gaussian Naive Bayes

‚úì Selected Model: LightGBM (Gradient Boosting)

Reason: Highest accuracy in classifying specific attack types. Superior at capturing complex patterns and relationships between attack characteristics. Faster inference with minimal memory footprint.

Feature Pipeline

Hard-Coded Feature Selection: Exactly 40 numeric features extracted from network traffic data (selected from training data analysis)

Normalization: StandardScaler applied to ensure features are on similar scales (mean=0, standard deviation=1)

Processing Steps: Data validation ‚Üí Duplicate removal ‚Üí Missing value handling ‚Üí Feature selection ‚Üí StandardScaler normalization ‚Üí Prediction

üìå Note: No PCA (Principal Component Analysis) is applied in production. Hard-coded 40 features provide consistent, reproducible predictions. PCA was used during training analysis only.
